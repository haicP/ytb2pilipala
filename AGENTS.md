# 全局 Agent 规则

## 指令优先级
1. 当前会话中用户的明确要求
2. 仓库自身规则、文档与约定
3. 本 AGENTS.md
4. 相关 skill 流程定义

## 默认原则

### 最短路径与并行轻重分流
- 默认采用"满足质量要求的最短路径"
- 默认先判断任务是否适合并行；适合则优先并行，不适合再串行
- 能直接完成并验证的，不升级为更重流程
- 能用轻量 planning 解决的小任务，不升级为重文档流程

### 轻量任务默认策略
- 轻量任务：单文件修改、明确 bug 修复、配置调整、小测试补充
- 提问：轻量任务首次最多问 1 个关键问题；中任务优先一次性给 2 到 3 个方案
- 默认授权边界：当前分支内可默认修改直接相关的应用代码、测试、局部文档
- 以下操作必须确认：删除文件、大规模重构、shared contract、根配置、CI、数据库变更

### 流程升级 / 降级
- 升级：影响边界超出初始判断、涉及公共 API / schema / 持久化 / 并发、需求不清晰
- 降级：改动局部边界清晰、不涉及共享核心逻辑、验证直接

## 任务分流模型

### 只读任务
- 分析、解释、架构说明、代码阅读——直接处理

### 实现任务
- 轻量版 planning 至少明确：目标、边界、风险、验证方式

## 并行开发总控

### 子代理派发策略
- 任何 `spawn_agent` 调用都必须显式设置 `model` 与 `reasoning_effort`，禁止省略，禁止依赖默认继承，禁止交由平台自行决定。
- 除非存在更高优先级指令的强制覆盖，子代理模型仅允许使用 `gpt-5.5` 与 `gpt-5.4`。
- 子代理默认使用 `gpt-5.5`。
- 仅当任务以代码实现、测试修复、局部重构、单模块阅读与分析为主，且不需要复杂跨模块推理时，才可使用 `gpt-5.4`。
- `reasoning_effort` 仅允许使用 `high` 或 `xhigh`，严格禁止使用 `medium`、`low` 或任何更低等级，也不得省略。
- 推理等级选择规则：
  - `high`：常规实现、测试补充、局部修复、单模块分析
  - `xhigh`：复杂排障、跨模块重构、架构设计、多约束权衡、高不确定性任务
- 若任务复杂度存在明显歧义，一律上调为 `xhigh`，不得下调到 `medium` 及以下。
- 派发前应先判断是否确有委派价值；一旦决定派发，必须在回复中简要说明本次选择该模型与推理等级的原因。
- 子代理继续派发下级子代理时，必须完整继承本节约束，不得自行放宽。
- 若因平台限制或更高优先级规则被迫偏离本策略，必须在回复中明确说明触发原因、实际使用的模型与推理等级，以及偏离范围。

### 调度闭环
- spawn_agent -> 记录 agent_id -> wait_agent 等待 -> 收集结果 -> close_agent
- 不要用普通命令等待替代子代理等待语义

### 并行准入
- 适合并行：可拆 2-4 个边界清晰、可独立验证的子任务
- 不适合并行：改动集中在 1-2 个核心文件、涉及 shared contract、需要独立分支隔离

### 收尾整合
- 子任务完成后必须统一收尾
- 至少包括：汇总改动、检查冲突面、运行最终验证、输出 merge plan

## 测试策略
- Level 0：定向验证（局部、低风险）
- Level 1：回归测试（中小修复）
- Level 2：TDD（新功能、共享逻辑、高风险改动）
- Level 3：Code Review
- Level 4：Completion Verification

## commit 规范
- 格式：`<type>(scope): <summary>`
- 常用 type：feat / fix / refactor / docs / test / chore
- summary：中文、动词开头、<=50 字、不加句号

- 不使用非 Git 工具操作 .git
- 不将密钥、API Key 硬编码进源码
- 数据库访问使用参数化查询
- 不用不可信输入拼接 shell 命令或 SQL

# 项目目标：YouTube 自动下载、翻译并上传 B 站工作台

## 产品定位
- 实现一个自动下载 YouTube 视频、生成与翻译字幕、合成中文配音、生成投稿信息，并上传到 B 站的视频处理工作台。
- 工作台页面需仿照 `http://127.0.0.1:8096/dashboard/` 的信息架构、密度、导航习惯与操作反馈；正式实现前必须先打开并观察该页面，不得凭空臆测布局。
- 页面是实际可用的操作台，不做营销落地页，不做空洞介绍页。

## 核心视频处理工作流
1. 导入任务：支持导入本地视频或提交 YouTube 链接。
2. 下载资产：下载 YouTube 视频与缩略图。
3. 音频处理：从视频中提取音频。
4. 字幕生成：基于音频生成源语言字幕。
5. 字幕翻译：将源语言字幕翻译为目标语言字幕，默认目标语言为中文。
6. 配音合成：基于翻译字幕生成中文配音。
7. 音视频同步：将原视频、翻译配音和字幕进行时间轴对齐，并支持同步预览播放。
8. 投稿信息生成：生成 B 站标题、简介、标签等投稿元数据。
9. 视频上传：上传处理后的视频到 B 站。
10. 字幕上传：上传 B 站字幕并与视频稿件关联。

## 工作台能力边界
- 任务列表：展示任务状态、来源、标题、进度、当前步骤、错误信息、创建时间与操作入口。
- 任务详情：展示工作流步骤、日志、输入资产、生成资产、字幕与配音预览、投稿元数据和上传结果。
- 手动干预：允许用户在上传前编辑标题、简介、标签、字幕与封面。
- 失败恢复：每个步骤应能记录失败原因，并支持从失败步骤重试。
- 状态追踪：长耗时任务应使用明确的进度、阶段状态和日志反馈，避免只显示全局 loading。
- 凭据管理：LLM Key、B 站凭据、cookie、token 等不得硬编码到源码，不得提交到仓库。

## 环境管理
- 本地开发使用 conda 管理 Python 环境，环境名为 `ytb2bb`；运行 Python、pytest 等命令时优先使用 `rtk conda run -n ytb2bb ...`。
- 项目应提供或维护 conda 环境说明，例如 `environment.yml` 或等价文档。
- 最终运行环境使用 Docker 封装。
- Docker 环境必须包含运行所需系统依赖，至少包括 `yt-dlp`、`ffmpeg` 与 Python 运行时。
- 本地环境与 Docker 环境的关键版本差异必须可追踪，避免“本地能跑、容器不能跑”。

## UI / UX 开发规范
- 页面设计必须使用 `ui-ux-pro-max` skill。
- 开始任何工作台页面实现或视觉改动前，必须执行该 skill 的设计系统检索流程，优先生成并持久化设计系统：
  - `python3 .codex/skills/ui-ux-pro-max/scripts/search.py "SaaS dashboard video workflow automation bilibili youtube translation professional" --design-system --persist -p "YouTube to Bilibili Workbench" --page "dashboard"`
- 若项目技术栈明确，应补充对应 stack 指南检索；未明确时默认使用 `html-tailwind`。
- UI 风格应偏向高密度、专业、可扫描的 SaaS/运营工作台。
- 避免营销式 hero、装饰性卡片堆叠、无关插画、单一色相铺满页面。
- 操作按钮应使用明确图标与短文本；状态、错误、重试、上传结果必须可见。
- 表格、任务流、预览区、日志区、编辑区应有稳定尺寸与响应式约束，避免加载或内容变化导致布局跳动。
- 前端实现完成后，必须进行桌面与移动视口检查，确认无文字溢出、重叠、遮挡与空白关键区域。

## 后端与任务执行规范
- 工作流步骤必须显式建模，避免把完整流程写成不可恢复的单个长函数。
- 每个步骤应产出结构化状态：`pending`、`running`、`success`、`failed`、`skipped`。
- 长任务应持久化关键中间产物路径、步骤耗时、错误摘要和可重试上下文。
- 外部命令执行必须使用参数数组或安全封装，禁止拼接不可信 shell 字符串。
- 文件路径、YouTube 链接、标题、简介、标签等用户输入必须校验。
- 生成内容上传前必须允许用户预览和编辑。

## 验证要求
- 修改页面时至少进行一次本地构建或类型检查。
- 修改任务流程、外部命令调用、上传逻辑时必须补充或更新定向测试。
- 涉及 `yt-dlp`、`ffmpeg`、LLM、B 站上传等外部依赖时，应优先提供 mock / dry-run 验证路径。
- 完成前必须运行与改动范围匹配的验证命令，并在最终回复中说明验证结果。
